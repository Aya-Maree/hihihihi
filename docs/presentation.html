2Workflow & Agent Logic

7-step conditional planning workflow

1
Intake — Extract event type, date, guest count, budget, venue from natural language
2
Clarification — Identify gaps; ask targeted follow-up questions (dietary, accessibility)
3
Retrieval — Build enriched RAG query; fetch top-5 KB chunks from ChromaDB
4
Conflict Detection — LLM + KB flags budget, timeline, dietary conflicts before planning
5
Planning — Generate full cited narrative plan with quantities, costs, timelines
6
Validation — User adjustments answered with RAG; context updated dynamically
7
Artifact Generation — Gemini + KB produces all 3 structured documents

Conditional branching examples:

  • Budget < $8/person → conflict flag, block planning until resolved
  • Event < 7 days away → urgency alert, immediate action list
  • Dietary restrictions → dietary KB retrieved automatically
  • Children attending → children's activities added to plan
  • Conflicts detected → workflow pauses at Step 4, waits for resolution

Key property: the system will not generate a plan until all detected conflicts are resolved — it asks the user to make a decision, just like a real consultant would.

2Memory & State Tracking

Persistent EventContext across all turns

EventContext (JSON, in-memory session)

event_type: "birthday party"
event_date: "2026-04-25"
guest_count_estimated: 20
budget_total: $300.00
budget_allocated: $285.50
venue_type: "home"
dietary_restrictions: ["vegetarian"]
has_children: true
detected_conflicts: []
workflow_state: "validation"
pending_tasks: [...]
  • Extracted automatically from natural language — no manual form required
  • Persists across all 7 steps and unlimited chat turns in the same session
  • Updated dynamically — if user says "change guests to 15", the context updates and the plan re-generates with the new values
  • Enriches every RAG query — retrieval is always context-aware, not generic
  • Drives conditional logic — workflow branches based on what's in the context (conflicts, dietary flags, has_children, etc.)

ChatHistory also stored per session — last 50 messages, passed to Gemini as conversation context so multi-turn dialogue feels natural and coherent.

Total time: ~2 minutes from first message to downloadable planning package. All responses cite their source documents.

🧠 How Gemini drives decisions

At each step, the LLM receives: (1) the full EventContext JSON, (2) the current workflow state, (3) the top-5 retrieved KB chunks. It generates a response AND returns a structured JSON metadata block with any context updates, detected conflicts, and whether to advance the workflow.

🔄 How context updates work

Every user message is first passed through an extraction function. Detected updates (e.g. guest_count: 15) are applied to EventContext before the step handler runs. The next step sees the updated state.

⛔ Conflict gate

The system cannot advance past Step 4 while conflicts exist. It stays in the conflict/clarification loop until the user resolves each issue — preventing bad plans from being generated.

📎 Citations as evidence

Every response cites the source document(s) used. This grounds all recommendations in the knowledge base and makes outputs auditable — not AI hallucination.
